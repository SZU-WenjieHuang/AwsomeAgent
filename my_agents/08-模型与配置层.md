```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                      多 Agent 场景下的模型配置                                   │
└─────────────────────────────────────────────────────────────────────────────────┘

当前设计：每个 Agent 实例独立的模型配置
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   AgentManager                                                                  │
│   ├── _user_agents: dict[str, dict[str, BaseAgent]]                            │
│   │   # user_id → {agent_type → Agent}                                         │
│   │                                                                             │
│   └── get_or_create_agent(user_id, agent_type)                                 │
│       ├── 创建对应类型的 Agent 实例                                            │
│       ├── 每个 Agent 有自己的 active_llm_config                                │
│       └── 每个 Agent 可以独立切换模型                                          │
│                                                                                 │
│   示例：                                                                        │
│   ┌─────────────────────────────────────────────────────────────────────────┐  │
│   │ user_123:                                                               │  │

│   │   research: ResearchAgent(active_llm_config=GLM5_LLM)                  │  │
│   │   code_review: CodeReviewAgent(active_llm_config=KIMI_LLM)             │  │
│   └─────────────────────────────────────────────────────────────────────────┘  │

│   优势：                                                                        │
│   ✅ 不同 Agent 类型可以使用最适合的模型                                        │
│   ✅ 每个 Agent 实例可以独立切换模型                                            │
│   ✅ 配置隔离，互不影响                                                         │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

是否需要改造？
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   结论：✅ 模型配置层已完全满足多 Agent 需求，无需改造                           │
│                                                                                 │
│   原因分析：                                                                    │
│                                                                                 │
│   1. ActiveLLMConfig 是实例级别的配置                                          │
│      ┌─────────────────────────────────────────────────────────────────────┐   │
│      │ • 每个 Agent 实例在 __init__ 时传入自己的 active_llm_config         │   │

│      │ • 运行时可以通过 switch_model() 独立切换                             │   │
│      └─────────────────────────────────────────────────────────────────────┘   │
│                                                                                 │
│   2. LLM 工厂是纯函数，无状态                                                   │
│      ┌─────────────────────────────────────────────────────────────────────┐   │
│      │ • create_llm_from_active_config() 根据传入配置创建 LLM              │   │
│      │ • 不依赖全局状态，线程安全                                             │   │
│      │ • 多个 Agent 可以同时调用，互不干扰                                    │   │
│      └─────────────────────────────────────────────────────────────────────┘   │
│                                                                                 │
│   3. 配置持久化已支持按 Agent 类型隔离                                          │
│      ┌─────────────────────────────────────────────────────────────────────┐   │
│      │ 方案 A：每个 Agent 类型独立的 config.yaml（推荐）                   │   │

│      │   ├── config.yaml                    # 通用配置                     │   │
│      │   ├── taskclaw-config.yaml           # TaskClaw 专用                │   │
│      │   └── research-config.yaml           # Research 专用                │   │
│      │                                                                     │   │
│      │ 方案 B：统一 config.yaml，按 agent_type 分组（当前）                │   │
│      │   model:                                                             │   │
│      │     taskclaw: { name: "qwen3-coder:30b", ... }                      │   │
│      │     research: { name: "glm-5:cloud", ... }                          │   │
│      └─────────────────────────────────────────────────────────────────────┘   │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

## 总结

| 问题 | 回答 |
|------|------|
| **模型配置层是否通用？** | ✅ 是的，完全通用 |
| **多 Agent 是否需要改造？** | ✅ 不需要，已天然支持 |
| **与存储/沙箱层的关系？** | 三者都是底层基础设施，设计理念一致：通过实例隔离而非类型感知 |

**核心设计原则**：

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   底层基础设施三层（存储、沙箱、模型配置）都遵循相同的设计原则：                 │
│                                                                                 │
│   1. 完全通用：不感知 Agent 类型                                                │
│   2. 实例隔离：每个 Agent 实例独立创建和管理资源                                │
│   3. 标识隔离：通过 user_id / session_id / thread_id 实现多租户                 │
│                                                                                 │
│   这种设计的优势：                                                              │
│   ✅ 职责清晰，每层只负责单一职责                                               │
│   ✅ 易于扩展，新增 Agent 类型无需修改底层代码                                    │
│   ✅ 易于测试，每层可以独立测试                                                 │
│   ✅ 易于维护，修改不影响其他层                                                 │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```
