## 并发网关层问题解答

---

### 问题 0：为什么需要这个调度网关层？

**核心原因：AI Agent 任务执行是"重"操作**

```
传统 Web 请求                    AI Agent 任务
    │                                │
    ▼                                ▼
┌─────────────┐              ┌─────────────────┐
│  毫秒级响应  │              │  秒级~分钟级执行  │
│  轻量计算    │              │  LLM 推理        │
│  可高并发    │              │  工具调用        │
└─────────────┘              │  文件读写        │
                             │  沙箱执行        │
                             └─────────────────┘
```

**没有网关会发生什么？**

```
场景：10 个用户同时发起任务

❌ 没有网关：
┌────────────────────────────────────────────────────┐
│  用户A × 3 任务  ──┐                                │
│  用户B × 2 任务  ──┼──► 服务器同时执行 10 个任务     │
│  用户C × 5 任务  ──┘                                │
│                                                    │
│  结果：                                             │
│  • 内存爆炸（每个 Agent 加载 LLM）                   │
│  • CPU 100%，系统卡死                               │
│  • 单用户占用所有资源，其他用户饥饿                   │
│  • 同一会话并发冲突（状态错乱）                       │
└────────────────────────────────────────────────────┘

✅ 有网关：
┌────────────────────────────────────────────────────┐
│  用户A × 3 任务  ──► 排队，最多 3 个并发             │
│  用户B × 2 任务  ──► 排队，等待用户A 释放            │
│  用户C × 5 任务  ──► 排队，等待全局资源              │
│                                                    │
│  结果：                                             │
│  • 系统稳定，资源可控                                │
│  • 公平调度，无饥饿                                  │
│  • 会话串行，状态一致                                │
└────────────────────────────────────────────────────┘
```

**一句话总结**：网关层是 AI 任务的"交通指挥官"，防止系统被重负载任务压垮。

### 问题 1：什么是 Handler？

**Handler = 执行任务的函数/回调**

```python
# Handler 就是一个异步函数，封装了"如何执行这个任务"

# 示例 1：用户请求的 Handler
async def stream_handler():
    agent = await agent_manager.get_or_create_agent(user_id)
    async for event in agent.stream_with_state(message, thread_id):
        yield event  # 流式输出

# 示例 2：后台任务的 Handler
async def stream_collector_handler():
    agent = await agent_manager.get_or_create_agent(user_id)
    full_content = []
    async for event in agent.stream_with_state(message, thread_id):
        if event.get("type") == "content":
            full_content.append(event.get("content"))
    return "".join(full_content)  # 收集完整结果
```

**Handler 不是唯一标识，而是执行逻辑**：

```
任务请求
    │
    ▼
┌─────────────────────────────────────────┐
│  command_id = "abc123"    ← 唯一标识     │
│  handler = stream_handler ← 执行逻辑     │
│  user_id = "user_A"                      │
│  session_id = "sess_001"                 │
└─────────────────────────────────────────┘
    │
    ▼
  Gateway.handle_task(command_id, handler, ...)
```


### 问题 3：什么时候标记空闲/忙碌？

**状态标记时机**：

```python
# ─────────────────────────────────────────────────────
# 忙碌：任务开始执行时
# ─────────────────────────────────────────────────────

async def handle_task(...):
    # ① 立即标记忙碌
    self.activity_monitor.mark_busy(user_id)
    
    # ② 获取锁（可能需要等待）
    await global_lane.acquire()
    await user_lane.acquire()
    await session_lock.acquire()
    
    # ③ 执行任务
    result = await handler()
    
    # ─────────────────────────────────────────────────
    # 空闲：任务完成后
    # ─────────────────────────────────────────────────
    
    # ④ 释放锁
    session_lock.release()
    user_lane.release()
    global_lane.release()
    
    # ⑤ 标记空闲
    self.activity_monitor.mark_idle(user_id)
    
    return result
```

**状态图**：

```
         mark_busy()
    ┌────────────────┐
    │                │
空闲 ────► 忙碌 ────► 空闲
    │                │
    │   mark_idle()  │
    └────────────────┘
         (任务完成)
```

**流式响应的特殊处理**：

```python
# 流式响应不能立即标记空闲，需要等前端消费完

async def wrapped_generator():
    try:
        async for item in gen:
            yield item  # 持续输出...
    finally:
        # 前端消费完毕后才标记空闲
        session_lock.release()
        user_lane.release()
        global_lane.release()
        self.activity_monitor.mark_idle(user_id)  # ← 这里才标记
```

### 问题 4：周期性任务 vs 空闲任务

#### 周期性任务

**定义**：固定时间间隔执行，不管系统是否繁忙

```markdown
<!-- schedule.md 示例 -->

## daily_report
- type: periodic      ← 周期性
- handler: stream_collector
- prompt: "生成每日工作报告，汇总今天的任务完成情况"
- output: "./reports/daily_{date}.md"
- enabled: true
```

**执行时机**：

```
时间轴：
├── 00:00 执行 ──► 生成报告
├── 01:00 执行 ──► 生成报告
├── 02:00 执行 ──► 生成报告
│   ...
└── 24:00 执行 ──► 生成报告

不管系统是否繁忙，到点就执行
```

#### 空闲任务

**定义**：只在系统空闲时执行，不打扰用户

```markdown
<!-- schedule.md 示例 -->

## cleanup_logs
- type: idle         ← 空闲时
- handler: stream_collector
- prompt: "清理 7 天前的过期日志文件"
- enabled: true
```

**执行时机**：

```
时间轴：
├── 10:00 用户在执行任务 ──► 不执行（系统忙碌）
├── 10:30 用户在执行任务 ──► 不执行（系统忙碌）
├── 11:00 用户空闲 5 分钟 ──► ✅ 执行清理任务
├── 11:30 用户又发起任务 ──► 不执行（系统忙碌）
└── 12:00 用户空闲 5 分钟 ──► ✅ 执行清理任务

只有系统空闲超过阈值才执行
```

#### 对比

| 特性 | periodic | idle |
|------|----------|------|
| **执行条件** | 时间间隔到达 | 系统空闲 + 时间间隔 |
| **使用场景** | 定时报、定时同步 | 清理、整理、优化 |
| **对用户影响** | 可能抢占资源 | 不影响用户 |
| **例子** | 每日报告、定时备份 | 日志清理、缓存整理 |

#### 为什么需要后台调度？

```
用户主动请求                     后台自动任务
```

```python
async def execute_scheduled_task(user_id, task_name):

    handler = factory.create_for_scheduled_task(

        handler=handler,
        user_id=user_id,

    if instruction.output and result:
        output_path.write_text(result)
```

### 二、核心组件详解

#### 1. CommandQueue - 三级并发控制

**设计原理**：漏桶模型 + 信号量控制

```python
# 三级并发控制结构
┌─────────────────────────────────────────────────────────────┐
│                     全局信号量                      │
│                    max_concurrent = 50                       │
│                    限制系统总并发数                           │
└────────────────────────────┬────────────────────────────────┘
                             │
          ┌──────────────────┴──────────────────┐
          │                                     │
          ▼                                     ▼
┌─────────────────────┐              ┌─────────────────────┐
│  用户信号量 (User)   │              │  后台信号量│
│  max_concurrent = 3 │              │  max_concurrent = 4  │
│  每用户独立实例      │              │  独立资源池          │
└──────────┬──────────┘              └─────────────────────┘
           │
           ▼
┌─────────────────────┐
│  会话锁   │
│  asyncio.Lock       │
│  确保同会话串行      │
└─────────────────────┘
```

**核心数据结构**：

```python
@dataclass
class Lane:
    """请求通道，包含并发控制的信号量"""
    name: str
    semaphore: asyncio.Semaphore  # 信号量控制并发数
    max_concurrent: int
    current_count: int = 0        # 当前活跃数

@dataclass
class SessionLock:
    """会话锁，确保同一会话的任务串行执行"""
    session_id: str
    lock: asyncio.Lock
    last_access: datetime         # 用于清理空闲锁
```

**锁获取顺序**（防止死锁）：

```python
# 按顺序获取锁：全局 -> 用户 -> 会话
global_acquired = await self._global_lane.acquire()
user_acquired = await user_lane.acquire()
acquired = await session_lock.acquire()

# 释放顺序相反：会话 -> 用户 -> 全局
session_lock.release()
user_lane.release()
self._global_lane.release()
```

**解决的问题**：
| 问题 | 解决方案 |
|------|----------|
| 系统过载 | 全局信号量限制总并发数 |
| 单用户占用过多资源 | 用户级信号量限制 |
| 同一会话并发冲突 | 会话锁串行化 |
| 后台任务抢占资源 | 独立后台资源池 |

#### 2. ActivityMonitor - 活跃度监控

**设计原理**：引用计数 + 时间戳判断

```python
class ActivityMonitor:
    _global_active_count: int = 0        # 全局活跃计数
    _global_last_active: datetime        # 最后活跃时间
    _user_activities: Dict[str, UserActivity]  # 用户活动状态
    idle_threshold: int = 5              # 空闲阈值（秒）
```

**状态转换**：

```
           mark_busy()
    空闲 ──────────────► 忙碌
     ▲                      │
     │                      │
     └──────────────────────┘
           mark_idle()
     (active_count == 0 且 elapsed >= threshold)
```

**核心方法**：

```python
def is_idle(self) -> bool:
    """判断系统是否空闲"""
    if self._global_active_count > 0:
        return False
    elapsed = (datetime.now() - self._global_last_active).total_seconds()
    return elapsed >= self.idle_threshold
```

**解决的问题**：
| 问题 | 解决方案 |
|------|----------|
| 判断何时执行后台任务 | `is_idle()` 判断系统空闲 |
| 用户级空闲判断 | `is_user_idle(user_id)` |
| 监控系统负载 | `get_stats()` 返回活跃状态 |

#### 3. TaskScheduler - 后台任务调度

**设计原理**：时间轮 + 持久化注册表

```python
@dataclass
class TaskEntry:

    task_name: str
```

**两种调度模式**：

```python
# 1. periodic - 周期性任务
if task.mode == "periodic":
    elapsed = (now - last_run).total_seconds()
    if elapsed >= task.interval:
        should_execute = True

# 2. idle - 空闲时任务
elif task.mode == "idle":
    if self.activity_monitor.is_idle():
        elapsed = (now - last_run).total_seconds()
        if elapsed >= task.interval:
            should_execute = True
```

**调度器主循环**：

```python
async def _scheduler_loop(self):

        await self._check_and_execute_tasks()
        await asyncio.sleep(1)  # 每秒检查一次
```

**设计原理**：Markdown 配置文件解析


**解析流程**：

```python
def _parse_schedule(self, content: str) -> List[TaskInstruction]:
    # 1. 按 ## 分割任务块
    task_blocks = re.split(r'^##\s+', content, flags=re.MULTILINE)
    
    # 2. 解析每个块的属性
    for block in task_blocks:
        lines = block.split('\n')
        task_name = lines[0].strip()
        # 解析 - key: value 格式
        props = self._parse_properties(lines[1:])
        
    # 3. 创建 TaskInstruction 对象
    return instructions
```

#### 5. Gateway - 统一入口

**核心方法 `handle_task()`**：

```python
async def handle_task(
    self,
    user_id: str,
    session_id: str,
    handler: Callable[[], Awaitable[T]],
):
    """
    处理用户任务请求的完整流程：
    1. 标记活跃
    2. 获取三级锁
    3. 执行 handler
    4. 释放锁
    5. 标记空闲
    """
    # 1. 标记活跃
    self.activity_monitor.mark_busy(user_id)
    
    # 2. 获取资源（三级锁）
    user_lane = await self.command_queue._get_user_lane(user_id)
    session_lock = await self.command_queue._get_session_lock(session_id)
    
    await self._global_lane.acquire()
    await user_lane.acquire()
    await session_lock.acquire()
    
    try:
        # 3. 执行 handler
        result = await handler()
        
        # 流式响应特殊处理
        if isinstance(result, AsyncGenerator):
            return self._wrap_streaming_response(result, ...)
        return result
        
    finally:
        # 4. 释放锁（逆序）
        session_lock.release()
        user_lane.release()
        self._global_lane.release()
        
        # 5. 标记空闲
        self.activity_monitor.mark_idle(user_id)
```

**流式响应包装**：

```python
def _wrap_streaming_response(self, gen, user_id, session_lock, user_lane):
    """包装流式响应，确保消费完毕后释放锁"""
    async def wrapped_generator():
        try:
            async for item in gen:
                yield item
        finally:
            # generator 消费完毕后释放锁
            session_lock.release()
            user_lane.release()
            self._global_lane.release()
            self.activity_monitor.mark_idle(user_id)
    
    return wrapped_generator()
```

### 三、核心设计模式

#### 1. 分层资源控制

```
请求 → 全局检查 → 用户检查 → 会话检查 → 执行
         ↓           ↓           ↓
      信号量(50)   信号量(3)    Lock(1)
```

#### 2. 资源池隔离

```
┌─────────────────────────────────────────┐
│              用户请求资源池               │
│  global_max_concurrent = 50             │
│  user_max_concurrent = 3                │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│              后台任务资源池               │
│  background_max_concurrent = 4          │
│  （独立，不抢占用户资源）                 │
└─────────────────────────────────────────┘
```

#### 3. 防死锁设计

- **锁获取顺序**：全局 → 用户 → 会话（固定顺序）
- **锁释放顺序**：会话 → 用户 → 全局（逆序）
- **超时机制**：`acquire_timeout` 防止无限等待
