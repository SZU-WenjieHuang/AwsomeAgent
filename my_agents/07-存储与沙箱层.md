### 2. CheckpointerManager - 会话状态持久化

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                        CheckpointerManager 架构                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

设计目的
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   LangGraph 的 Checkpointer 负责：                                              │
│   • 保存 Agent 执行过程中的完整状态                                             │
│   • 支持会话恢复（从任意快照恢复）                                               │
│   • 支持时间旅行（回溯历史状态）                                                 │
│   • 支持 interrupt/resume（暂停/恢复）                                          │
│                                                                                 │
│   CheckpointerManager 的职责：                                                  │
│   • 为每个用户创建独立的 SQLite 数据库                                          │
│   • 管理 AsyncSqliteSaver 的生命周期                                            │
│   • 提供统一的初始化和关闭接口                                                   │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

存储位置
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   user-data/{user_id}/.db/sessions.db                                          │
│                                                                                 │
│   与 SessionManager 共享同一个数据库文件：                                       │
│   ┌─────────────────────────────────────────────────────────────────────────┐  │
│   │                                                                         │  │
│   │   sessions 表 (SessionManager 管理)                                     │  │
│   │   └── 会话元数据：id, user_id, model, status, created_at, updated_at   │  │
│   │                                                                         │  │
│   │   checkpoint 表 (Checkpointer 管理)                                     │  │
│   │   └── 会话快照：thread_id, checkpoint_id, checkpoint (JSON), metadata  │  │
│   │                                                                         │  │
│   │   checkpoint_blobs 表                                                   │  │
│   │   └── 大对象存储（如图片）                                               │  │
│   │                                                                         │  │
│   │   checkpoint_writes 表                                                  │  │
│   │   └── 写入记录                                                          │  │
│   │                                                                         │  │
│   └─────────────────────────────────────────────────────────────────────────┘  │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

核心代码
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   class CheckpointerManager:                                                   │
│       def __init__(self, backend_root_dir: str):                               │
│           self.backend_root_dir = Path(backend_root_dir)                       │
│           self._checkpointer: Optional[AsyncSqliteSaver] = None                │
│           self._context = None                                                 │
│                                                                                 │
│       async def initialize(self) -> AsyncSqliteSaver:                          │
│           db_path = self.backend_root_dir / ".db" / "sessions.db"              │
│           self._context = AsyncSqliteSaver.from_conn_string(str(db_path))      │
│           self._checkpointer = await self._context.__aenter__()                │
│           return self._checkpointer                                            │
│                                                                                 │
│       async def close(self) -> None:                                           │
│           await self._context.__aexit__(None, None, None)                      │
│                                                                                 │
│       def get_checkpointer(self) -> Optional[AsyncSqliteSaver]:                │
│           return self._checkpointer                                            │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

在 Agent 中的使用
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   # TaskClawAgent 初始化时                                                      │
│   self._checkpointer_manager = CheckpointerManager(backend_root_dir)           │
│   checkpointer = await self._checkpointer_manager.initialize()                 │
│                                                                                 │
│   # 创建 DeepAgent 时传入                                                       │
│   self.agent = create_deep_agent(                                              │
│       model=llm,                                                               │
│       checkpointer=checkpointer,  # ← 注入 Checkpointer                        │
│       backend=backend,                                                         │
│       ...                                                                      │
│   )                                                                            │
│                                                                                 │
│   # 执行时使用 thread_id 关联会话                                               │
│   config = {"configurable": {"thread_id": session_id}}                         │
│   async for event in self.agent.astream(input, config=config):                 │
│       ...                                                                      │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 3. E2BSandbox - E2B 云端沙箱

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                          E2BSandbox 架构                                        │
└─────────────────────────────────────────────────────────────────────────────────┘

E2B 简介
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   E2B (Code Interpreter) 是一个云端代码执行平台：                               │
│   • 提供隔离的虚拟机环境                                                        │
│   • 支持多种预配置模板（如 code-interpreter-v1）                                │
│   • 内置 Python、Bash 等运行时                                                  │
│   • 按使用时长计费                                                              │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

类结构
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   E2BSandbox(BaseSandbox)                                                      │
│   ├── _sandbox: E2B Sandbox 实例                                               │
│   ├── _workdir: str = "/home/user"                                             │
│   ├── _default_timeout: int = 1800 (30分钟)                                    │
│   │                                                                             │
│   ├── execute(command, timeout) → ExecuteResponse                              │
│   │   └── 调用 sandbox.commands.run(command, cwd=workdir, timeout=...)         │
│   │                                                                             │
│   ├── download_files(paths) → List[FileDownloadResponse]                       │
│   │   └── 调用 sandbox.files.read(path, format="bytes")                        │
│   │                                                                             │
│   ├── upload_files(files) → List[FileUploadResponse]                           │
│   │   └── 调用 sandbox.files.write(path, content)                              │
│   │                                                                             │
│   └── kill() → None                                                            │
│       └── 调用 sandbox.kill()                                                  │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

E2BProvider - 生命周期管理
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   E2BProvider(api_key)                                                         │
│   │                                                                             │
│   ├── get_or_create(sandbox_id=None, template="code-interpreter-v1")           │
│   │   ├── 如果 sandbox_id 存在：Sandbox.connect(sandbox_id)                    │
│   │   ├── 如果 sandbox_id 为空：Sandbox.create(template=template)             │
│   │   ├── _wait_until_ready() 轮询直到沙箱就绪                                  │
│   │   ├── _validate_runtime_tools() 检查 bash 和 python3 可用                  │
│   │   └── 返回 E2BSandbox 实例                                                 │
│   │                                                                             │
│   └── delete(sandbox_id)                                                       │
│       └── Sandbox.kill(sandbox_id)                                             │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

使用流程
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   from taskclaw.sandbox import E2BProvider                                     │
│                                                                                 │
│   # 1. 创建 Provider                                                           │
│   provider = E2BProvider(api_key="your-e2b-api-key")                           │
│                                                                                 │
│   # 2. 创建沙箱                                                                 │
│   backend = provider.get_or_create(template="code-interpreter-v1")             │
│   print(f"Sandbox ID: {backend.id}")                                           │
│                                                                                 │
│   # 3. 执行命令                                                                 │
```

## 四、多 Agent 改造影响分析

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                      多 Agent 改造对存储与沙箱层的影响                           │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 1. OSSStorageService：无需改造 ✅                                               │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│   原因：                                                                        │
│   • OSS 路径已经包含 user_id，天然支持多用户隔离                                │
│   • 可以增加 agent_type 到路径中，实现按 Agent 类型隔离                         │
│                                                                                 │
│   可选优化：                                                                    │
│   user_data/{user_id}/{agent_type}/sessions/...                                │
│   user_data/{user_id}/{agent_type}/reports/...                                 │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 2. CheckpointerManager：无需改造 ✅                                             │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│   原因：                                                                        │
│   • Checkpointer 只关心 thread_id（session_id）                                │
│   • 不同 Agent 的会话使用不同的 thread_id，天然隔离                             │
│   • 可以在 session_id 中包含 agent_type                                        │
│                                                                                 │
│   可选优化：                                                                    │
│   session_id 格式: sess_{agent_type}_{uuid}                                    │
│   例: sess_taskclaw_abc123, sess_research_def456                               │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 3. Sandbox：无需改造 ✅                                                         │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│   原因：                                                                        │
│   • 沙箱是 Agent 级别的资源，每个 Agent 实例可以有自己的沙箱                    │
│   • 不同 Agent 类型可以使用相同的沙箱实现                                       │
│   • 也可以根据 Agent 类型选择不同的沙箱类型                                     │
│                                                                                 │
│   配置示例：                                                                    │
│   taskclaw:                                                                    │
│     sandbox: { type: local }                                                   │
│   research:                                                                    │
│     sandbox: { type: e2b, template: code-interpreter-v1 }                      │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 4. OSSSyncBackend：需要小改造 ⚠️                                                │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│   改造点：                                                                      │
│   • 增加可选的 agent_type 参数                                                  │
│   • OSS 路径映射包含 agent_type                                                 │
│                                                                                 │
│   当前路径：                                                                    │
│   user_data/{user_id}/sessions/{session_id}/...                                │
│                                                                                 │
│   改造后路径：                                                                  │
│   user_data/{user_id}/{agent_type}/sessions/{session_id}/...                   │
│                                                                                 │
│   代码变更：                                                                    │
│   def __init__(self, inner, storage, user_id, session_id, agent_type="taskclaw"):│
│       self._agent_type = agent_type                                            │
│                                                                                 │
│   def _to_oss_key(self, virtual_path):                                         │
│       return f"user_data/{self._user_id}/{self._agent_type}/{path}"            │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

总结
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   组件                │ 改造需求    │ 说明                                     │

│   OSSStorageService   │ ✅ 无需    │ 路径已包含 user_id                      │
│   CheckpointerManager │ ✅ 无需    │ thread_id 隔离已足够                    │
│   VirtualLocalShell   │ ✅ 无需    │ 每个 Agent 实例独立                     │
│   E2BSandbox          │ ✅ 无需    │ 每个 Agent 实例独立                     │
│   DaytonaSandbox      │ ✅ 无需    │ 每个 Agent 实例独立                     │

│   核心结论：                                                                    │
│   存储与沙箱层设计良好，天然支持多 Agent 架构，几乎不需要改造。                 │
```

### 总结

|------|------|------|------------------|
| **存储层** | OSSStorageService | 阿里云 OSS 文件存储 | ✅ 无需改造 |
| **存储层** | CheckpointerManager | LangGraph 会话状态持久化 | ✅ 无需改造 |

存储与沙箱层的设计非常清晰，采用了装饰器模式和协议抽象，天然支持多 Agent 架构。



这是一个很好的架构设计问题。让我分析一下存储和沙箱层在多 Agent 场景下的适配策略：

## 存储与沙箱层：通用 vs 专用

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                      两种设计策略对比                                            │
└─────────────────────────────────────────────────────────────────────────────────┘

策略一：保持通用（推荐 ✅）
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   设计原则：存储和沙箱层完全通用，不感知 Agent 类型                              │
│                                                                                 │
│   OSS 路径结构：                                                                │
│   user_data/{user_id}/sessions/{session_id}/...                                │
│                      ↑                                                          │
│                   不包含 agent_type，session_id 已足够区分                       │
│                                                                                 │
│   沙箱实例：                                                                    │
│   每个 Agent 实例独立创建沙箱，沙箱不感知 Agent 类型                             │
│                                                                                 │
│   优势：                                                                        │
│   ✅ 架构简单，职责清晰                                                         │
│   ✅ 存储层专注于存储，沙箱层专注于执行                                          │
│   ✅ 新增 Agent 类型无需修改存储/沙箱代码                                        │
│   ✅ 易于测试和维护                                                             │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

策略二：Agent 专用（不推荐 ❌）
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   设计原则：存储和沙箱层感知 Agent 类型，按类型隔离                              │
│                                                                                 │
│   OSS 路径结构：                                                                │
│   user_data/{user_id}/{agent_type}/sessions/{session_id}/...                   │
│                      ↑                                                          │
│                   包含 agent_type，物理隔离                                      │
│                                                                                 │
│   沙箱实例：                                                                    │
│   不同 Agent 类型使用不同的沙箱配置/模板                                         │
│                                                                                 │
│   劣势：                                                                        │
│   ❌ 增加复杂性，存储层需要感知业务逻辑                                          │
│   ❌ 新增 Agent 类型需要修改存储/沙箱代码                                        │
│   ❌ 违反单一职责原则                                                           │
│   ❌ 难以维护（路径规则分散在各处）                                              │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

## 推荐方案：保持通用

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                      推荐架构：分层职责清晰                                      │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 第一层：API 层（感知 Agent 类型）                                               │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│   FastAPI 路由                                                                  │
│   ├── /api/chat (agent_type 参数)                                              │
│   ├── /api/sessions?agent_type=xxx (按类型过滤)                                │
│   └── /api/agents/{agent_type}/...                                             │
│                                                                                 │
│   职责：                                                                        │
│   • 接收 agent_type 参数                                                        │
│   • 路由到对应的 Agent 实例                                                     │
│   • 按 Agent 类型过滤会话列表                                                   │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│ 第二层：AgentManager（感知 Agent 类型）                                         │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│   AgentManager                                                                  │
│   ├── _user_agents: dict[str, dict[str, BaseAgent]]                            │
│   │   # user_id → {agent_type → Agent}                                         │
│   │                                                                             │

│                                                                                 │

│   • 管理 user_id + agent_type → Agent 的映射                                    │

                                    │
```

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                      关键设计点：隔离与标识                                      │
└─────────────────────────────────────────────────────────────────────────────────┘

1. Session ID 设计（包含 Agent 类型信息）
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   方案 A：前缀格式（推荐）                                                      │
│   session_id = f"sess_{agent_type}_{uuid[:12]}"                                │
│   例：                                                                          │
│   • sess_taskclaw_abc123def456                                                  │
│   • sess_research_abc123def456                                                  │
│   • sess_code_review_abc123def456                                               │
│                                                                                 │
│   优点：                                                                        │
│   • 从 session_id 可直接识别 Agent 类型                                         │
│   • Checkpointer 使用 thread_id = session_id，天然隔离                          │
│   • 无需修改存储层代码                                                          │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

2. 文件路径设计（无需包含 Agent 类型）
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   OSS 路径：                                                                    │
│   user_data/{user_id}/sessions/{session_id}/report.md                          │
│                                                                                 │
│   不需要：                                                                      │
│   user_data/{user_id}/{agent_type}/sessions/{session_id}/report.md             │
│                                                                                 │
│   原因：                                                                        │
│   • session_id 已包含 agent_type 信息                                           │
│   • 路径层级越少越好，易于维护                                                   │
│   • 列出用户文件时，通过 session_id 前缀过滤即可                                 │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

3. 沙箱实例管理
┌─────────────────────────────────────────────────────────────────────────────────┐
│                                                                                 │
│   AgentManager 中的管理：                                                       │
│   ┌─────────────────────────────────────────────────────────────────────────┐  │
│   │                                                                         │  │
│   │   _user_agents: dict[str, dict[str, BaseAgent]]                        │  │
│   │   {                                                                     │  │
│   │       "user_123": {                                                     │  │

│   │                                                                         │  │

│                                                                                 │
│   每个 Agent 实例：                                                             │
│   • 有自己的沙箱后端（可以是不同类型）                                          │
```

## 总结

| 层级 | 是否感知 Agent 类型 | 职责 | 改造需求 |
|------|-------------------|------|---------|
| **API 层** | ✅ 感知 | 接收 agent_type 参数，路由到对应 Agent | 需要增加 agent_type 参数 |
| **AgentManager** | ✅ 感知 | 管理 user_id + agent_type → Agent 映射 | 需要改为二级字典 |
| **BaseAgent** | ❌ 不感知 | 实现 Agent 特定业务逻辑，使用通用资源 | 无需改造 |
| **存储层** | ❌ 不感知 | 提供通用存储能力 | ✅ 保持通用，无需改造 |
| **沙箱层** | ❌ 不感知 | 提供通用执行环境 | ✅ 保持通用，无需改造 |

**核心原则**：存储和沙箱层应该保持通用，通过 `user_id` 和 `session_id`（包含 agent_type 信息）实现隔离，而不是让底层感知业务逻辑。这样设计：

1. **职责清晰**：存储层只负责存储，沙箱层只负责执行
2. **易于扩展**：新增 Agent 类型无需修改存储/沙箱代码
3. **易于维护**：路径规则集中在一处（session_id 生成）
4. **可测试**：底层组件可以独立测试
