## 什么是 Agent 里的上下文工程？

### 一句话定义

> **上下文工程（Context Engineering）= 在每一步推理时，把"恰好正确"的信息放进模型的上下文窗口里。**

Andrej Karpathy 的原话：
> "Context engineering is the delicate art and science of filling the context window with just the right information for the next step."

---

### 和 User Memory 的关系

User Memory 是上下文工程的**一个数据来源**，但上下文工程远不止 Memory。

可以用这张图来理解：

```
上下文工程（Context Engineering）
├── 写入（Write）         ← 把信息存到上下文窗口外
│   ├── Scratchpad（草稿本）：当前任务的中间状态
│   └── Memory（记忆）：跨会话的长期信息 ← User Memory 在这里
│
├── 选择（Select）        ← 把需要的信息拉进上下文窗口
│   ├── 从 Memory 里召回相关记忆
│   ├── 从工具结果里选关键内容
│   └── 从 Skill 库里按需加载（懒加载）
│
├── 压缩（Compress）      ← 减少 token 占用
│   ├── 历史对话摘要（Claude Code 的 auto-compact）
│   ├── 工具输出截断/摘要
│   └── Observation Masking（旧 tool 输出替换为占位符）
│
└── 隔离（Isolate）       ← 把上下文拆开，避免互相干扰
    ├── 多 Agent 各自独立上下文窗口
    └── 沙箱隔离（代码执行结果不直接进上下文）
```

---

### 为什么 Agent 比普通 LLM 更需要上下文工程

普通 LLM 是**单轮**的，上下文就是用户问题 + 系统 prompt，相对简单。

Agent 是**持续决策系统**，上下文会随着任务推进不断累积：

```
Turn 1: system prompt + user query
Turn 2: + tool call result（搜索结果）
Turn 3: + tool call result（文件内容）
Turn 4: + tool call result（数据库查询）
...
Turn N: 上下文已经几万 token，模型开始"发懵"
```

这会导致四类问题（LangChain 的分类）：
- **Context Poisoning**：幻觉内容进入上下文，污染后续推理
- **Context Distraction**：无关信息太多，模型注意力被分散
- **Context Confusion**：上下文里有矛盾信息，模型不知道信谁
- **Context Clash**：不同来源的信息互相冲突

---

### 上下文工程的四大操作

| 操作 | 做什么 | 例子 |
|------|--------|------|
| **Write（写）** | 把信息存到窗口外 | 把用户偏好写入 Memory，把中间结果写入 Scratchpad |
| **Select（选）** | 按需拉入窗口 | 召回相关记忆、懒加载 Skill、RAG 检索 |
| **Compress（压）** | 减少 token | 历史摘要、工具输出截断、Observation Masking |
| **Isolate（隔）** | 拆分上下文 | 多 Agent 各自独立窗口、沙箱隔离 |

---

### User Memory 在其中的位置

**User Memory 属于 Write + Select 的组合：**

- **Write 阶段**：把用户的偏好、身份、历史行为写入长期记忆存储（跨会话持久化）
- **Select 阶段**：每次对话开始时，从 Memory 里召回与当前任务相关的记忆，注入上下文

**但 User Memory 只是上下文工程的一小块**，上下文工程还要管：
- 当前任务的 Skill 怎么加载（懒加载 vs 全量）
- 历史 tool 输出怎么压缩（摘要 vs Masking）
- 多 Agent 之间上下文怎么隔离
- system prompt 和 user input 怎么分层（角色隔离）

---

### 核心原则

> **"给模型正确的信息"，而不是"给模型更多的信息"**

上下文工程的本质是**信息的精准调度**，不是堆砌。堆得越多，模型越容易发懵，这是 Agent 工程里最容易踩的坑。
