## 一、问题对照表（先建立框架）

| 失效模式 | 核心根因 | 工程对策方向 |
|---|---|---|
| **Context Rot** | Softmax 注意力权重被稀释，信噪比下降 | 主动压缩 / 裁剪，减少 token 总量 |
| **Context Drift** | 目标措辞在压缩中偏移，正反馈加速漂移 | 结构化摘要锚定目标，保护原始意图 |
| **复合精度崩溃** | 每步独立误差在多步管道中指数级叠加 | 分阶段校验，中间状态持久化 |
| **上下文中毒** | 早期幻觉被当成"事实"写入上下文 | 工具输出质量标记，事前剪枝 |
| **Lost-in-the-Middle** | 关键信息落在注意力盲区（中间段） | 保护 Head + Tail，把重要内容放边缘 |
| **无限循环** | 上下文膨胀 → 压缩 → 信息丢失 → 重新搜索 | 硬上限 + 重复动作检测 |

## 二、Hermes Agent — 工业级多层防御

Hermes 是四个项目里上下文管理最系统、最完整的。核心是 [context_compressor.py](/Users/huangwenjie/Documents/AwsomeAgent/AwsomeAgent/2-hermes-agent/agent/context_compressor.py) + [context_engine.py](/Users/huangwenjie/Documents/AwsomeAgent/AwsomeAgent/2-hermes-agent/agent/context_engine.py)。

### 2.1 压缩算法：五阶段流水线

```
消息列表 → [Phase 1] 工具输出剪枝 → [Phase 2] 确定边界 → 
[Phase 3] LLM 结构化摘要 → [Phase 4] 拼装压缩后消息 → [Phase 5] 修复孤儿 tool pair
```

**Phase 1 — 工具输出廉价剪枝（无 LLM 调用）**

```python
# 只剪超过 200 字符的旧 tool result，用占位符替换
_PRUNED_TOOL_PLACEHOLDER = "[Old tool output cleared to save context space]"
# 保护 tail_token_budget 以内的最近 N 条消息不被剪
if len(content) > 200:
    result[i] = {**msg, "content": _PRUNED_TOOL_PLACEHOLDER}
```

这一步直接对抗 **上下文中毒** 和 **无限循环**：把已经没有用的旧工具输出清空，不让它累积成噪声。

**Phase 2 — Head + Tail 双端保护**

```python
# 保护 Head：system prompt + 前 3 条消息（包含原始意图）
compress_start = self.protect_first_n  # 默认 3

# 保护 Tail：按 token 预算而非固定消息数
# tail_token_budget = threshold * target_ratio (默认 20K tokens)
compress_end = self._find_tail_cut_by_tokens(messages, compress_start)
```

保护 Head 直接对抗 **Context Drift**：原始 system prompt 和用户最初的目标，永远不会被压缩掉。保护 Tail 对抗 **Lost-in-the-Middle**：最近的决策和工具结果留在 recency 位置。

**Phase 3 — 结构化摘要 Prompt（对抗 Drift 的核心）**

```python
_template_sections = """
## Goal
[What the user is trying to accomplish]

## Constraints & Preferences
[User preferences, coding style, constraints, important decisions]

## Progress
### Done      # 已完成工作 → 不要重做
### In Progress
### Blocked

## Key Decisions
[Important technical decisions and WHY they were made]

## Resolved Questions
[已经回答了的问题 → 不要再回答]

## Pending User Asks
[还没答的问题 → 新会话要处理]

## Relevant Files
## Remaining Work
## Critical Context
## Tools & Patterns
"""
```

这个结构化摘要模板是 Hermes 应对 **Context Drift** 最精妙的设计：

1. 明确 "Goal" 锚定原始意图，防止目标在摘要中措辞漂移
2. "Resolved vs Pending Questions" 分离：新 context 里的 Agent 不会去重答已解决的问题
3. "Remaining Work" 替代 "Next Steps"：避免模型把摘要当主动指令去执行

**Phase 5 — 孤儿 Tool Pair 修复**

```python
# 压缩后可能出现 tool_result 引用了不存在的 tool_call
# 用 stub result 填充，保持消息格式合法
patched.append({
    "role": "tool",
    "content": "[Result from earlier conversation — see context summary above]",
    "tool_call_id": cid,
})
```

### 2.2 迭代更新摘要（多次压缩的关键）

```python
if self._previous_summary:
    # 不是从头摘要，而是"更新上次摘要"
    prompt = f"""
    PREVIOUS SUMMARY: {self._previous_summary}
    NEW TURNS TO INCORPORATE: {content_to_summarize}
    
    PRESERVE all existing information that is still relevant.
    ADD new progress. Move items from "In Progress" to "Done".
    """
```

这里解决了一个很深的问题：**多次压缩的累积漂移**。如果每次压缩都从头生成摘要，前两次摘要的措辞差异就会累积，"转化价值" → "贡献" → "研究方法"的经典漂移就会发生。Hermes 把每次摘要当成一个增量更新，而不是重新总结。

### 2.3 三层触发机制

```yaml
compression:
  threshold: 0.50        # 达到 50% context 就触发（远在 overflow 之前）
  target_ratio: 0.20     # 压缩后目标：只保留 20% 的 tail
  protect_last_n: 20     # 至少保护最近 20 条消息
```

同时有三道防线：

| 防线 | 触发条件 | 作用 |
|---|---|---|
| Agent 自身压缩器 | 50% context | 主动防 Rot |
| Gateway 卫生检查 | 85% context | 跨会话兜底 |
| 硬上限 | 400 条消息 | 防死循环（断开 → 无 token 数据 → 无法压缩 → 死螺旋）|

```python
# 硬安全阀 —— 打破"API 断开 → 无 token 数据 → 无法压缩"的死螺旋
_HARD_MSG_LIMIT = 400
_needs_compress = (
    _approx_tokens >= _compress_token_threshold
    or _msg_count >= _HARD_MSG_LIMIT
)
```

### 2.4 Memory 系统与压缩协同

压缩前会触发 `flush_memories`：

```python
# 压缩前先让 memory provider 把重要信息持久化
self.flush_memories(messages, min_turns=0)
# 通知 memory manager 即将失去的上下文
if self._memory_manager:
    self._memory_manager.on_pre_compress(messages)
```

通过 `on_pre_compress` 钩子，各 memory provider 可以在消息被压缩掉之前提取并存入长期记忆（MEMORY.md 文件），实现"压缩不丢关键记忆"。

### 2.5 可插拔上下文引擎

```python
# context_engine.py — 抽象基类
class ContextEngine(ABC):
    def should_compress(self) -> bool: ...
    def compress(self, messages) -> messages: ...
    def get_tool_schemas(self) -> list: ...  # 引擎可以给 Agent 暴露工具！
```

设计非常工程化：默认用 `ContextCompressor`，但 LCM（Lossless Context Management）插件可以完全替换整个策略。

## 三、OpenClaw — 企业级多模态 Compaction

OpenClaw 的上下文管理体现了一个关键思想：**不同场景用不同策略**。

### 3.1 Compaction 模式选择

```typescript
// extensions.ts
if (resolveCompactionMode(params.cfg) === "safeguard") {
    setCompactionSafeguardRuntime(params.sessionManager, {
        maxHistoryShare: ...,        // 历史消息最多占 context 多少比例
        contextWindowTokens: ...,    // 总 context 窗口大小
        recentTurnsPreserve: ...,    // 保留最近几轮
        customInstructions: ...,     // 自定义摘要指令
        qualityGuardEnabled: ...,    // 是否开启摘要质量检测
        qualityGuardMaxRetries: ..., // 质量不达标最多重试几次
    });
}
```

**qualityGuard** 是个独特设计：摘要生成后会再检测质量，不达标就重试。直接对抗 Context Drift 中"摘要措辞偏移"的问题。

### 3.2 可插拔 CompactionProvider

```typescript
// plugins/compaction-provider.ts
interface CompactionProvider {
    id: string;
    label: string;
    summarize(params: {
        messages: unknown[];
        compressionRatio?: number;
        customInstructions?: string;
        previousSummary?: string;  // 支持迭代更新，和 Hermes 一样
    }): Promise<string>;
}
```

### 3.3 三步 Compaction 流程

```typescript
// compact.ts
// Step 1: 校验会话历史（sanitizeSessionHistory）
const prior = await sanitizeSessionHistory({ messages, ... });

// Step 2: 验证回放轮次（validateReplayTurns）
const validated = await validateReplayTurns({ messages: prior, ... });

// Step 3: 限制历史轮数（limitHistoryTurns）
const truncated = limitHistoryTurns(
    session.messages,
    getDmHistoryLimitFromSessionKey(params.sessionKey, params.config),
);

// Step 4: 修复 tool_use/tool_result 配对（和 Hermes 一样！）
const limited = sanitizeToolUseResultPairing(truncated, {
    erroredAssistantResultPolicy: "drop",
});
```

这里 `validateReplayTurns` 是一个非常工程化的设计：**重放验证**。把现有历史消息重新走一遍验证流程，确保消息结构合法，这对防止上下文中毒（前期错误被传播）非常有效。

### 3.4 Channel 级别的历史限制

```typescript
// 群组聊天的历史长度独立配置
const historyLimit = Math.max(
    0,
    msteamsCfg?.historyLimit ??
        cfg.messages?.groupChat?.historyLimit ??
        DEFAULT_GROUP_HISTORY_LIMIT,
);
```

OpenClaw 是多平台 Agent，在不同 Channel（Teams、Telegram、Slack等）上的历史长度策略不同。这体现了一个重要思想：**Context 管理必须结合业务场景**，不能全局一刀切。

## 四、LangGraph — 框架级的 pre_model_hook 机制

LangGraph 的策略不是内置一个固定压缩算法，而是**提供一个标准钩子，让用户自由实现**。

### 4.1 pre_model_hook — 模型调用前的截断钩子

```python
# chat_agent_executor.py
create_react_agent(
    model=...,
    tools=...,
    pre_model_hook=trim_messages_hook,  # 每次调用模型前先处理消息
)
```

`pre_model_hook` 的返回值结构：

```python
# 两种返回方式：
# 1. 覆写 messages（真正改变状态）
{
    "messages": [RemoveMessage(id=REMOVE_ALL_MESSAGES), *new_messages]
}

# 2. 只影响本次 LLM 输入，不改变状态
{
    "llm_input_messages": trimmed_messages
}
```

这两种方式的区别非常重要：
- 方式1：永久性修改状态，适合定期压缩
- 方式2：临时性输入处理，只影响当次推理，状态不变

### 4.2 `trim_messages` — LangChain 核心工具

```python
from langchain_core.messages.utils import trim_messages

trimmed = trim_messages(
    messages,
    max_tokens=4000,
    strategy="last",           # 保留最近的消息
    token_counter="approximate",
    start_on="human",          # 确保从 HumanMessage 开始（格式合法）
    include_system=True,       # 保护 SystemMessage
    allow_partial=False,       # 不切割单条消息
)
```

这里有几个细节直接对抗失效模式：
- `include_system=True`：永远保住 system prompt（对抗 Drift，保护原始约束）
- `start_on="human"`：确保消息序列从 Human 开始（防止格式错误导致 API 报错）
- `strategy="last"`：保留最近的消息（recency bias 是客观存在的，顺应它）

### 4.3 SummarizationMiddleware — 来自 LangChain 的标准中间件

LangGraph 项目本身的 prebuilt 库已经被 `langchain` 的 `SummarizationMiddleware` 取代（create_react_agent 已标注为 deprecated，推荐用 `langchain.agents.create_agent`）：

```python
# langchain/agents/middleware/summarization.py
class SummarizationMiddleware(AgentMiddleware):
    def before_model(self, state, runtime):
        messages = state["messages"]
        total_tokens = self.token_counter(messages)
        
        if not self._should_summarize(messages, total_tokens):
            return None  # 不处理，透传
        
        cutoff_index = self._determine_cutoff_index(messages)
        messages_to_summarize, preserved = self._partition_messages(messages, cutoff_index)
        
        # 生成摘要
        summary = self._create_summary(messages_to_summarize)
        
        # 返回：用摘要消息替换旧消息
        return {
            "messages": [
                RemoveMessage(id=REMOVE_ALL_MESSAGES),
                HumanMessage(f"Here is a summary of the conversation to date:\n\n{summary}"),
                *preserved_messages,
            ]
        }
```

触发条件支持三种：

```python
# tokens: 达到绝对 token 数
("tokens", 3000)
# messages: 达到消息数量
("messages", 50)  
# fraction: 达到模型 max_input_tokens 的比例（多模型场景自适应）
("fraction", 0.8)
```

`fraction` 类型是 LangGraph 生态对抗不同模型上下文长度差异的关键设计——同一套配置在 32K 和 128K 模型上自动缩放。

## 五、DeerFlow — 分层记忆 + 中间件管道

DeerFlow 的架构分成两个维度：**会话级压缩（Summarization）** + **跨会话长期记忆（Memory）**。

### 5.1 SummarizationMiddleware 配置

```yaml
# config.yaml
summarization:
  enabled: true
  model_name: gpt-4o-mini    # 用廉价模型做摘要（重要的成本优化！）
  trigger:
    - type: tokens
      value: 15564
    - type: messages
      value: 50
  keep:
    type: messages
    value: 10                 # 摘要后只保留最近 10 条
  trim_tokens_to_summarize: 15564  # 送给摘要模型的 token 上限
```

DeerFlow 的中间件管道有明确的执行顺序：

```
ThreadDataMiddleware
SandboxMiddleware
SummarizationMiddleware  ← 上下文管理在这里
TitleMiddleware
ClarificationMiddleware
```

这个顺序很重要：Summarization 在 Clarification（追问）之前——先把上下文压缩到合理大小，再决定是否需要追问用户。

### 5.2 长期记忆结构化存储

DeerFlow 实现了一个真正的分层记忆系统，对应不同时间粒度：

```python
# memory/updater.py 的 Memory 数据结构
{
    "user": {
        "workContext": {"summary": "..."},      # 工作背景
        "personalContext": {"summary": "..."},   # 个人偏好
        "topOfMind": {"summary": "..."},         # 当前关注点
    },
    "history": {
        "recentMonths": {"summary": "..."},      # 近期历史
        "earlierContext": {"summary": "..."},    # 更早历史
        "longTermBackground": {"summary": "..."}, # 长期背景
    },
    "facts": [                                   # 结构化事实
        {
            "id": "fact_abc123",
            "content": "User prefers Python over JavaScript",
            "category": "preference",
            "confidence": 0.9,
            "createdAt": "...",
        }
    ]
}
```

这里对抗 **Context Drift** 的关键设计是：
- 工作背景、个人偏好、当前关注点**分开存储**，不互相污染
- 结构化 facts 而非自然语言总结，避免摘要措辞漂移
- `confidence` 字段支持可信度分级（解决你面经里的"冲突消解"问题）
- `max_facts` 限制 + 按 confidence 排序保留 topK（防止 Memory 无限增长）

## 七、核心工程原则提炼

结合四个项目的实现，对抗 Context Rot / Drift 的工程原则可以总结为：

**1. 主动防御，远在 overflow 之前触发**
四个项目都在 50-80% context 时触发压缩，而不是等到 100% 报错。

**2. Head + Tail 双端锚定**
- Head（system prompt + 原始意图）：永远不压缩 → 对抗 Drift
- Tail（最近 N 条）：按 token budget 保护 → 对抗 Lost-in-the-Middle

**3. 摘要不是"总结"，是"结构化状态转移文档"**
Hermes 的结构化模板是最典型的案例：Goal / Resolved / Pending 的区分，让摘要成为新 context 的"状态快照"，而不是模糊的自然语言总结。

**4. 迭代更新 > 重新摘要**
多次压缩时，保存上次摘要作为基础增量更新，防止每次摘要都引入新的措辞漂移。

**5. 压缩前先剪枝（廉价 pre-pass）**
工具输出超过阈值就替换为占位符——无需 LLM 调用，直接消灭大量噪声 token。

**6. 孤儿 Tool Pair 必须修复**
压缩后必然出现 `tool_call` 和 `tool_result` 不匹配，必须有专门的 sanitize 步骤，否则 API 直接报错。

**7. 压缩 + 长期记忆双轨运行**
压缩处理会话内的上下文 Rot，长期记忆（MEMORY.md / memory.json）处理跨会话的目标漂移。两者不可互相替代。
以上就是四个项目在上下文管理上的完整工程实现对比。Hermes 在**主动压缩 + 迭代摘要**上最成熟；OpenClaw 在**多平台历史策略 + 质量守卫**上最精细；LangGraph 在**框架级可扩展钩子**上最灵活；DeerFlow 在**分层长期记忆 + 信号驱动更新**上最有特色。


## 八、压缩之前先存储
**1. 压缩不是孤立的 —— 压缩前必须先"存"**

为什么这一步必须存在？ 因为压缩是有损的。结构化摘要再好，也会丢失细节（具体的文件路径、具体的错误码、具体的用户措辞）。所以在压缩之前，必须给系统一个机会把这些细节持久化到压缩之外的地方（MEMORY.md、memory.json、向量库等）。

**2. 压缩解决的是"会话内"，记忆解决的是"跨会话"**
