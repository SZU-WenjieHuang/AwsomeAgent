# LLM 知识检索技术选型全景图

## 核心原则

**不是"什么技术最先进"，而是"什么技术最适合这种知识类型"。**

---

## 一、按知识类型分类的技术选型

### 1️⃣ **精确代码片段/函数定义**

**知识特征：**
- 需要精确匹配函数名、类名、变量名
- 关注语法结构而非语义
- 通常需要完整上下文（整个函数/类）

**推荐技术：Grep + AST 解析**

```python
# 场景：找到某个函数的定义
查询："getUserInfo 函数在哪里定义的？"

技术方案：
1. Grep 精确搜索：grep -r "def getUserInfo" .
2. AST 解析：提取完整函数体
3. 返回：文件路径 + 完整代码 + 调用位置

优势：
✅ 精确匹配，零误差
✅ 保留完整语法结构
✅ 支持正则表达式

劣势：
❌ 无法语义搜索（如"获取用户信息的方法"）
❌ 需要知道确切名称
```

**适用场景对比：**

| 查询类型 | Grep | RAG 向量检索 |
|---------|------|------------|
| "getUserInfo 在哪定义的" | ✅ 精确找到 | ⚠️ 可能不精确 |
| "获取用户信息的函数" | ❌ 无法匹配 | ✅ 语义理解 |
| "所有带 @deprecated 注解的函数" | ✅ 正则匹配 | ❌ 不适合 |

---

### 2️⃣ **文档/注释/Wiki 内容**

**知识特征：**
- 自然语言描述
- 语义相似但表述不同
- 需要理解上下文和背景知识

**推荐技术：RAG 向量检索**

```python
# 场景：理解某个模块的用途
查询："认证模块是如何工作的？"

技术方案：
1. 文档分块（按章节/语义）
2. Embedding 计算（如 bge-large）
3. 向量数据库检索（HNSW 索引）
4. 返回：最相关的文档片段 + 来源

优势：
✅ 语义理解（"认证" ≈ "鉴权" ≈ "authentication"）
✅ 支持模糊查询
✅ 可跨文档检索

劣势：
❌ 对专业术语不如关键词精确
❌ 需要 embedding 计算成本
```

**实现示例：**
```python
# LangChain RAG 流程
documents = PDFLoader("api-docs.pdf").load()
chunks = RecursiveCharacterTextSplitter().split_documents(documents)
vectors = OpenAIEmbeddings().embed_documents([c.page_content for c in chunks])
vectorstore = Chroma.from_documents(chunks, embeddings)

results = vectorstore.similarity_search("如何调用认证接口", k=5)
```

### 3️⃣ **代码语义理解（功能相似代码）**

**知识特征：**
- 需要理解代码的功能意图
- 不同实现方式做同一件事
- 查找"类似功能"的代码

**推荐技术：Code RAG + AST 语义嵌入**

```python
# 场景：找到类似的实现
查询："有没有其他处理文件上传的代码？"

技术方案：
1. 代码 AST 解析（提取函数签名、调用关系）
2. Code Embedding（如 CodeBERT、StarCoder）
3. 向量检索 + 语法树匹配

示例：
用户代码：
  def upload_file(file):
      validate(file)
      save_to_s3(file)
      update_db(file)

检索到的相似代码：
  def process_image(img):           ← 不同函数名
      validate_image(img)           ← 相同验证逻辑
      upload_to_oss(img)            ← 相同存储逻辑  
      save_metadata(img)            ← 相同数据库逻辑

优势：
✅ 理解功能意图
✅ 跨语言检索（Python → JavaScript）
✅ 找到最佳实践

劣势：
❌ 需要专门的代码 embedding 模型
❌ 对复杂逻辑理解有限
```

**专用工具：**

| 工具 | 特点 | 适用场景 |
|------|------|---------|
| **CodeBERT** | 微软开源，支持 6 种编程语言 | 代码相似度检索 |
| **StarCoder** | 大规模代码预训练模型 | 代码补全、检索 |
| **Tree-sitter** | 语法树解析 | 精确代码结构分析 |
| **Sourcegraph** | 代码搜索引擎 | 大规模代码库检索 |

### 4️⃣ **代码结构化理解（依赖/调用关系）**

**知识特征：**
- 函数调用链
- 类继承关系
- 模块依赖图
- 数据流追踪

**推荐技术：知识图谱 + AST**

```python
# 场景：理解代码架构和依赖
查询："getUserInfo 被哪些接口调用？"

技术方案：
1. AST 解析提取：
   - 函数定义
   - 函数调用
   - 类继承
   - 模块导入
2. 构建知识图谱：
   (getUserInfo, 被调用, API /users/profile)
   (getUserInfo, 调用, validateToken)
   (UserController, 继承, BaseController)
3. 图遍历查询

知识图谱结构：
┌──────────────┐      调用       ┌──────────────┐
│  /api/users  │ ──────────────→ │ getUserInfo  │
└──────────────┘                 └──────────────┘
                                         │
                                   调用    │
                                         ↓
                                 ┌──────────────┐
                                 │validateToken │
                                 └──────────────┘

优势：
✅ 精确的依赖关系
✅ 支持多跳查询（A→B→C）
✅ 可视化代码架构

劣势：
❌ 构建成本高（需要解析整个代码库）
❌ 对动态语言支持有限
❌ 无法理解语义
```

**实现示例：**
```python
# Neo4j 知识图谱查询
query = """
MATCH (api:Endpoint)-[:CALLS]->(func:Function)-[:CALLS]->(dep:Function)
WHERE func.name = 'getUserInfo'
RETURN api.name, func.name, dep.name
"""

# 结果：
# API: /api/users/profile
# Function: getUserInfo
# Dependencies: [validateToken, queryDatabase, formatResponse]
```

### 5️⃣ **API 文档 + 使用示例**

**知识特征：**
- 接口定义（路径、参数、返回值）
- 使用示例代码
- 错误码说明

**推荐技术：结构化检索（数据库）+ RAG 混合**

```python
# 场景：查询 API 用法
查询："创建订单接口怎么调用？"

技术方案：
1. 结构化数据库存储：
   - endpoint: POST /api/orders
   - parameters: {userId, items, paymentMethod}
   - example: {代码示例}
2. 混合检索：
   - 关键词匹配端点名
   - 向量检索理解意图

数据库结构：
{
  "endpoint": "/api/orders",
  "method": "POST",
  "description": "创建新订单",
  "parameters": {
    "userId": "string, required",
    "items": "array, required"
  },
  "example": "POST /api/orders {...}",
  "embedding": [0.123, -0.456, ...]  # 用于语义检索
}

查询流程：
用户："怎么创建订单？"
  ↓
向量检索 → 匹配到 /api/orders
  ↓
返回：接口定义 + 参数说明 + 示例代码
```

### 6️⃣ **错误日志/调试信息**

**知识特征：**
- Stack trace
- 错误码
- 已知的解决方案

**推荐技术：关键词检索 + 模板匹配**

```python
# 场景：排查错误
查询："NullPointerException at UserService.java:45"

技术方案：
1. 错误日志解析（提取关键信息）
2. 模板匹配（已知错误库）
3. 返回：相似错误 + 解决方案

匹配示例：
输入错误：
  java.lang.NullPointerException
    at com.example.UserService.getUserInfo(UserService.java:45)

匹配到知识库：
  错误模式：UserService.*\.java:\d+
  原因：用户 ID 为 null 时未做校验
  解决方案：添加 null check
  修复 PR：#1234

优势：
✅ 精确匹配错误模式
✅ 直接定位问题
✅ 提供修复方案

劣势：
❌ 无法处理未见过的错误
❌ 需要维护错误知识库
```

## 二、完整技术选型矩阵

| 知识类型 | 查询示例 | 首选技术 | 备选技术 | 核心优势 |
|---------|---------|---------|---------|---------|
| **精确代码定位** | "getUserInfo 在哪定义" | **Grep + AST** | 代码搜索工具 | 精确匹配 |
| **文档理解** | "认证模块如何工作" | **RAG 向量检索** | Wiki 搜索 | 语义理解 |
| **相似代码查找** | "类似的文件上传代码" | **Code RAG** | GitHub Copilot | 功能相似性 |
| **依赖关系** | "谁调用了这个函数" | **知识图谱 + AST** | IDE 导航 | 结构化关系 |
| **API 用法** | "创建订单接口" | **结构化 DB + RAG** | Swagger/OpenAPI | 精确 + 灵活 |

| **历史经验** | "之前如何处理类似需求" | **会话记忆 + RAG** | Git commits | 可追溯 |

## 三、Agent 集成策略

### 场景驱动的工具路由

```python
class KnowledgeRetrievalAgent:
    """
    根据查询意图自动选择检索技术
    """
    
    def route_query(self, query):
        """
        查询分类 → 技术路由
        """
        intent = self.classify_intent(query)
        
        if intent == "code_location":
            # "xxx 在哪定义"、"找到 xxx 函数"
            return self.grep_search(query)
            
        elif intent == "understand_concept":
            # "xxx 是什么"、"如何工作"
            return self.rag_search(query)
            
        elif intent == "find_similar":
            # "类似的代码"、"参考实现"
            return self.code_similarity_search(query)
            
        elif intent == "dependency":
            # "谁调用了"、"依赖关系"
            return self.graph_query(query)
            
        elif intent == "api_usage":
            # "怎么调用"、"接口参数"
            return self.api_search(query)
            
        elif intent == "error_debug":
            # "错误信息"、"异常堆栈"
            return self.error_pattern_match(query)
    
    def classify_intent(self, query):
        """
        使用 LLM 或规则分类查询意图
        
        规则示例：
        - 包含 "在哪"、"定义" → code_location
        - 包含 "如何"、"是什么" → understand_concept
        - 包含 "类似"、"参考" → find_similar
        - 包含 "调用"、"依赖" → dependency
        """
```

## 五、技术组合最佳实践

### 1. **Grep + RAG 混合**（代码 + 文档）

```python
# 场景：既需要精确代码，又需要理解背景
query = "订单超时如何处理？"

# 策略：
grep_results = grep("order.*timeout|timeout.*order")  
# → 找到 5 个相关文件

rag_results = vector_search("订单超时处理")
# → 找到 3 个相关文档

# 合并返回：代码片段 + 文档说明
```

### 2. **知识图谱 + RAG**（结构 + 语义）

```python
# 场景：既要理解依赖关系，又要理解设计意图
query = "为什么用 Kafka 而不是 REST？"

# 策略：
graph_results = query_graph("(订单服务, 使用, Kafka)")
# → 找到架构决策记录

rag_results = vector_search("Kafka vs REST 对比")
# → 找到技术选型文档

# 合并返回：依赖关系 + 决策理由
```

### 3. **AST + 向量检索**（语法 + 语义）

```python
# 场景：找到功能相似但实现不同的代码
query = "有没有其他验证邮箱的方法？"

# 策略：
ast_results = parse_functions("validate.*email|email.*valid")
# → 找到所有邮箱验证函数

vector_results = code_embedding_search("邮箱验证")
# → 找到语义相似的函数

# 合并返回：语法结构 + 功能相似度
```

## 六、总结

### 核心原则

| 问题 | 答案 |
|------|------|
| **Grep 过时了吗？** | ❌ 没有！精确匹配场景无可替代 |
| **RAG 能替代一切吗？** | ❌ 不能！结构化关系、精确位置需要其他技术 |
| **知识图谱有必要吗？** | ✅ 依赖关系、架构理解场景非常有用 |
| **最佳方案是什么？** | ✅ 根据查询类型**动态路由**到不同技术 |

### 技术选型口诀

```
精确定位用 Grep，
语义理解用 RAG，
依赖关系用图谱，
相似代码用 CodeRAG，
API 查询用结构化，
错误排查用模板，
最佳实践用 Wiki，
综合运用靠 Agent。
```

**记住：** 没有银弹，只有组合拳。关键是根据知识类型和查询意图，选择最合适的工具。
