## 重试时间加上Jitter，避免雷群效应

### 什么是 Jitter（抖动）？

**Jitter** 就是在重试等待时间上**加一个随机偏移量**，让每次重试的时间点不完全一致。

---

### 雷群效应（Thundering Herd）是什么？

先理解问题本身：

> 假设有 **1000 个客户端**同时请求同一个 LLM API，服务器在 `T=0` 时刻返回了 **429 Rate Limit** 错误。

**没有 Jitter 的情况（纯指数退避）：**

```
T=0     → 所有 1000 个客户端同时收到 429
T=1s    → 所有 1000 个客户端同时发起第 1 次重试 → 服务器再次被打垮 → 再次 429
T=2s    → 所有 1000 个客户端同时发起第 2 次重试 → 服务器再次被打垮 → 再次 429
T=4s    → 所有 1000 个客户端同时发起第 3 次重试 → ...
```

这就是**雷群效应**：一群客户端像"雷"一样，整整齐齐地同时冲击服务器，形成周期性的流量洪峰，服务器永远恢复不了。

---

### 加了 Jitter 之后：

```typescript
// OpenClaw 的 retry.ts 实现
const baseDelay = Math.min(minDelayMs * 2 ** attempt, maxDelayMs);
// jitter = 0.1，即 ±10% 的随机偏移
const jitterRange = baseDelay * jitter;
const actualDelay = baseDelay + (Math.random() * 2 - 1) * jitterRange;
```

**加了 Jitter 的情况：**

```
T=0     → 所有 1000 个客户端同时收到 429
T=1s    → 客户端们在 [0.9s ~ 1.1s] 之间随机分散重试
           → 服务器每个时刻只收到约 50 个请求，压力分散
T=2s    → 客户端们在 [1.8s ~ 2.2s] 之间随机分散重试
           → 服务器平稳恢复
```

---

### 具体数字对比

| 客户端 | 无 Jitter 第1次重试 | 有 Jitter（±10%）第1次重试 |
|--------|-------------------|--------------------------|
| 客户端 A | 1000ms | 1000ms × (1 + 0.07) = **1070ms** |
| 客户端 B | 1000ms | 1000ms × (1 - 0.03) = **970ms** |
| 客户端 C | 1000ms | 1000ms × (1 + 0.09) = **1090ms** |
| 客户端 D | 1000ms | 1000ms × (1 - 0.08) = **920ms** |

原本 1000 个客户端**同一毫秒**打过去，加了 Jitter 后分散到 **200ms 的时间窗口**内，服务器压力瞬间降低 **~5 倍**。

---

**一句话总结**：Jitter 就是给重试时间加随机盐，让"整齐的军队"变成"散漫的游客"，避免集体冲击服务器。
