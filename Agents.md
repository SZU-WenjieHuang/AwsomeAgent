# Agent 工程师面经整理

本文档整理了腾讯、懂车帝、小红书、阿里等公司的AI Agent开发岗位面试题目及参考答案。

---

## 一、tx

### 1. 项目里是把 skill 直接塞进 system prompt 的，如果 skill 太多，占用上下文窗口太大，怎么处理？ done

**问题分析：**
- 所有 skill 都直接嵌入 system prompt 会导致界面空间占用大
- 候选选择时噪音高
- 模型在选择 skill 时容易混淆

**解决方案：**
- 使用外部 skill 注册表，system prompt 中只保留最小规则和调用协议，按需动态注入完整 skill 描述
- 常见做法是先做一层 skill routing（通过规则、分类模型或向量搜索筛选 top-k skills），然后注入描述、参数 schema 和 few-shot 示例
- 对于长 skill 文本，分成摘要版和完整版，先注入摘要，模型决定调用后再加载详细说明
- 这样可以显著压缩 token

### 2. 如果两个 skill 实际内容不一样，但是 description 很相似，导致 agent 每次加载错 skill，怎么解决？done

**解决思路：**
- 当 description 相似时，单靠自然语言匹配很容易误召回
- 给 skill 增加更强的判别信息，而不是继续堆模糊描述

**具体步骤：**
1. **结构化定义升级**：将 skill 从纯文本描述升级为结构化定义，包括适用场景、禁止场景、输入参数、返回格式、前置条件、失败条件、调用示例
2. **两阶段选择**：先调 topk，再让模型根据参数约束和示例做 rerank
3. **增加负样本示例**：明确告诉模型哪些问题不应该触发该 skill
4. **执行前校验**：执行前做 schema 校验和 rule check，如果模型选错，执行中断，防止误调用

**语义相似时的处理：**
- 如果两个 skill 定义语义确实接近，建议在系统设计层合并为一个统一的 skill，内部用参数路由，而不是在 prompt 里制造歧义

### 3. Agent 设计里你觉得最重要的部分是什么？

**核心观点：上下文工程**

Agent 不是一个单轮问答系统，而是一个持续决策系统。模型能否稳定执行，取决于信息是如何被组织和优先级的：

- 规则、证据、历史状态、优先级如何被结构化呈现
- 上下文工程做好了，模型才能成为一个可控制的执行者——理解目标、约束、工具、历史和输出格式
- 做不好，就会出现目标漂移、工具误用、记忆污染、推理错误

**核心原则：** "给模型正确的信息"，而不是"给模型更多的信息"

### 4. 上下文工程有哪些要注意的点？ done

1. **角色分离与上下文隔离**：明确区分 system、developer、user、tool output、memory 等角色，外部文本和系统规则严格隔离，不要混用

2. **信息过滤与上下文裁剪**：只保留当前任务相关的内容，历史对话、搜索结果、工具文档中与当前任务无关的全部排除，避免模型认知过载

3. **格式控制与结构化输出**：工具输出用 JSON，任务状态用独立文本段，历史用摘要而非原文。尽量用结构化格式代替自然语言描述

4. **可信度层次与来源分层**：
   - 系统规则 > 用户输入 > 工具结果 > 模型预测 > 知识证据 > 未验证生成

5. **长任务管理**：长任务中要做文本压缩和过期过滤，避免状态漂移和旧信息干扰新决策

### 5. 你怎么设计 Agent 的长期记忆写回策略、衰减策略和冲突消解？

**核心观点：**
长期记忆不能等价于"把所有历史都存起来"。真正可用的长期记忆必须回答三个问题：什么值得写回、什么时候应该忘、冲突信息怎么处理。

**写回策略：**
- 只保存高价值、稳定的信息（用户偏好、身份属性、长期目标、反复出现的业务事实）
- 结构化抽象而非原文存储，例如将"用户更喜欢用 Python" → 存入 profile 字段，而非存整句

**衰减策略：**
- 有时效性和权重：短期热点可以高优先级，但随时间衰减
- 稳定的长期偏好可以是低频更新但不轻易删除

**冲突消解：**
- 结合时间戳、来源可信度、置信度进行判断
- 新的高置信信息直接覆盖旧的低置信内容
- 低置信内容只能作为备选，不能直接覆盖

**更严格的做法：**
将长期记忆设计为事件流 + 物化视图，所有记忆先作为事件写入，再异步聚合到 profile，支持审计和回滚。

### 6. 你怎么做并行 Tool Calling，既提升吞吐又保证一致性与可回放？

**核心观点：**
并行 Tool Calling 的核心不是"同时调用多个 API"，而是解决依赖关系、状态一致性、结果合并和失败回滚。

**关键原则：**
- 只有独立的工具才能并行执行（如天气查询和汇率查询可以同时进行）
- 有依赖关系的工具必须串行（如 Tool B 依赖 Tool A 的输出，则 B 必须等 A 完成）

**状态管理：**
- 工具不能直接修改共享状态，而是将结果写入临时观察区
- 由协调器统一合并成完整的 DAG
- 每次调用需要记录：trace_id、step_id、输入参数、输出结果、耗时

**失败处理：**
- 失败应该以单个工具粒度重试，而不是所有结果都丢弃
- 有副作用的操作（发消息、创建订单、提交工单）需要幂等机制，加上 nonce 或时间戳，防止重试造成业务数据重复

### 7. 了解主流的 Agent 设计吗？

**主流架构：**

1. **ReAct**：reason + act，模型思考、调用工具、观察结果，继续思考完成任务

2. **Plan-and-Execute**：先规划动作序列，再按步骤执行。比 ReAct 更稳定，适合长任务

3. **Router 模式**：先做任务分类，再路由给不同的子 Agent 或工具链

4. **Workflow + Agent**：固定流程里嵌局部自主决策，常见于线上业务场景

5. **Supervisor 模式**：由主 Agent 拆解任务并分派给多个子 Agent

6. **Blackboard 模式**：多个 Agent 通过共享状态协作

**注意：** 纯点对点自由协作在线上极少见，因为难以管理

### 8. MCP、A2A、普通 Function Calling 三者有什么本质区别，工程里怎么选？

**三者定义：**

- **Function Calling**：模型和宿主应用之间的接口，模型决定调用哪个函数，宿主执行并返回结果。本质是"调用一个函数"

- **MCP (Model Context Protocol)**：标准化资源/工具暴露协议，让外部能力能被模型客户端发现和调用。本质是"暴露一组标准能力"

- **A2A (Agent-to-Agent)**：Agent 间协作协议，负责任务/状态/结果的交换。本质是"多个智能体协作完成"

**选型建议：**
- 简单单应用工具使用 → Function Calling
- 多模型/IDE 共享工具资源 → MCP
- 复杂多 Agent 工作流，涉及任务路由和结果汇聚 → A2A

### 9. RAG 接到 Agent 里，一般怎么设计？

**核心观点：**
RAG 不应该固定为"先检索再回答"，而应该作为决策节点——判断是否需要外部知识，再决定查询重写、分解、召回、重排、生成的顺序。

**常见实现：**
- 将 RAG 封装成一个 tool，让 Agent 决定何时调用
- 结果返回前做预处理（去重、分块、排序、引用标记）
- 生成时只依据检索到的证据回答，没有证据就明确说不确定，这样才能真正降低幻觉

### 10. Tool Calling 一般怎么做，和普通函数调用有什么区别？

**本质区别：**
- **Tool Calling**：模型决策驱动的调用，宿主系统决定是否执行，需要额外的校验和编排层
- **普通函数调用**：开发者硬编码的调用，完全由开发者控制

**工程四层架构：**
1. **工具注册层**：管理工具的定义和元数据
2. **参数 schema 层**：定义和校验参数格式
3. **执行器层**：实际执行工具调用
4. **审计日志层**：记录调用链路

**生产环境注意事项：**
- 模型只能输出候选调用，服务端需要做白名单校验、参数校验、超时控制、限流、错误封装
- 不能直接给模型工具执行权限

### 11. Agent 的 Memory 一般怎么做？

**三种记忆类型：**

1. **短期记忆**：保留最近几轮对话和当前任务状态，确保连贯性

2. **长期记忆**：存储稳定事实，如用户偏好、身份属性、长期目标

3. **摘要记忆**：将历史对话压缩成关键事实，减少 token 使用

**实际部署策略：**
- 不存所有原始对话，只保留最近 N 轮原文
- 更早的内容压缩成摘要
- 关键句子或段落单独提取存储，按需召回
- 不要无脑全量拼接，而是按需召回

### 12. Prompt Injection 在 Agent 场景里怎么防？

**定义：**
Prompt Injection 是指在用户输入或外部文档中注入恶意指令，诱导模型忽略既定规则。

**Agent 场景的危险性：**
- 可能生成错误文本
- 误导工具调用
- 越权操作
- 甚至触发真实业务操作

**五层防护策略：**

1. **角色与优先级隔离**：外部知识不能覆盖系统规则
2. **文档即证据而非指令**：搜索结果/文档应被视为证据，而非可执行指令
3. **白名单与参数校验**：所有工具调用必须使用白名单名称和校验过的参数，高风险操作需要二次确认
4. **可追溯与可回滚**：输出和调用链可追溯，出错可回滚
5. **输入清洗与风险分级**：输入要清洗和风险分级，明显的越权或诱导内容直接拦截

**核心原则：**
"核心不是靠一句 prompt 防住，而是靠系统设计让模型即使被诱导，也没有越权能力"

### 13. 你怎么评估一个 Agent 系统的效果？

**四层评估框架：**

1. **回答质量层**：准确性、完整性、是否基于证据

2. **工具质量层**：选工具是否正确、参数是否正确、调用是否成功

3. **任务质量层**：一次会话是否完成目标、是否需要额外追问、是否需要人工接管

4. **系统质量层**：耗时、成本、token 消耗、错误率、稳定性

**RAG 系统额外评估：**
- 召回命中率
- rerank 效果
- 答案引用质量

**核心观点：**
Agent 不是单个模型的分数，而是整条链路的分数。

### 14. 如果线上发现 Agent 经常选错工具，你怎么排查？

**排查步骤：**

1. **检查 tool description 和 schema 设计**：边界是否模糊，参数能否区分场景

2. **检查 skill routing 层**：是否一次性塞了太多候选工具，导致噪音竞争

3. **检查 prompt 和 few-shot 示例**：是否给了误导性引导，system prompt 最近是否被改过

4. **回放 trace**：检查选到这个工具之前的步骤，历史记忆是否污染了当前判断

5. **检查用户输入**：是否存在矛盾，模型推理过程是否有二次校验

**核心观点：**
排查不是看最后一步，而是理解候选集的来源、模型为什么选它、执行前是否有校验。

### 15. 长任务场景里，怎么避免 Agent 无限推理或者死循环？

**解决策略：**

1. **使用状态机和终止条件**：
   - 强制限制最大推理步数、工具调用次数、token 消耗、执行时长
   - 明确完成、失败、用户反馈的触发条件

2. **增加重复动作检测**：
   - 如果连续几步重复使用同一工具且产出相似/无新证据
   - 强制终止并返回阶段性结论
   - 避免无意义的循环

### 2. 如果你的系统里也有高并发场景，你会怎么设计限流、降级和幂等？

**核心观点：**
平均延迟不是问题，真正的问题是少数慢请求可能拖垮整个请求链。

**多层限流设计：**

1. **入口层**：用户级、租户级限流
2. **任务调度层**：按任务队列隔离
3. **工具层**：并发池控制
4. **模型层**：token 预算和超时恢复

**幂等设计：**
- 写操作（发通知、审批、更新状态、创建工单）必须给每次执行一个 request_id 或 task_id
- 重复执行时能识别并返回历史结果，而不是让 Agent 失控重试
- 否则模型一次超时重发，后面会造成重复写入

### 3. 你怎么理解 Agent 项目的数据，和传统模型训练数据有什么不一样？

**核心观点：**
Agent 项目的"数据"不只是训练样本，它更像一整套运行时资产。

**Agent 数据包含：**
- 用户问题
- 标准答案
- 工具描述
- 执行轨迹
- 观察结果
- 失败日志
- 人工修正记录
- 最终是否完成任务

**与传统训练的区别：**
- 传统训练重视输入输出配对
- Agent 场景重视中间过程，用于理解模型行为（为什么选错工具、在哪里卡住、为什么会误读上下文）

**数据闭环：**
记录 thought proxy、tool call、observation、state transition 等中间步骤，用于优化和调试。

### 4. 如果让你优化一个 Agent 系统，你通常先从哪里下手？

**原则：**
不会一上来就换模型或改 prompt，而是先拆系统瓶颈。

**三类问题分类：**

1. **规划问题**：路径选错，比如两步的任务走了六步

2. **执行问题**：工具调用成本高、重试过多、失败转成功比

3. **观察问题**：模型输出的信息噪音大或散乱，导致每步判断不准确

**优化方向：**
- 性能优化：工具调用频次、上下文长度
- 效果优化：错误聚类分析，如分析态过早进入执行、证据不足导致无追问
- 有效优化通常是缩短错误链，而非孤立改进

### 5. 如果系统里需要接前端页面，你会怎么考虑 React 这类前端框架的接入方式？

**核心观点：**
前端接入不是简单做个聊天框。

**Agent 项目前端应承载的能力：**
- 过程可视化
- 步骤确认
- 局部纠错
- 结果回放

**React 等框架的优势：**
- 节点流展示
- 操作面板
- 实时状态更新

**Agent 多步执行场景：**
用户需要看到当前进行到哪一步、调了什么工具、拿到了什么结果。

**双层设计建议：**
1. **消息层**：处理自然语言交互
2. **任务层**：展示规划、步骤状态、风险确认、最终输出

实现对话模式（像聊天）+ 操作模式（像操作面板）双模式可用。

### 6. 你了解哪些 Agent 架构，分别适合什么问题？

**三种常见架构：**

1. **Single Agent + Tools**：适合边界清晰的任务（如问答、查数据、补全代码）

2. **Planner-Executor**：规划和执行分离，适合需要多步推理的长链决策

3. **Multi-Agent Collaboration**：不同 Agent 扮演不同角色（搜索、分析、审核、执行），适合复杂任务但控制难度大

**选型原则：**
根据任务复杂度选择架构，而不是堆 Agent。很多团队过早引入多角色，导致系统调试成本极高。

**何时用多 Agent：**
- 单 Agent 承载不了工作量的任务
- 子任务能力要求差异大的场景

### 7. 多 Agent 协作里最难解决的是什么？

**核心难点：**
不是让多个 Agent 会说话，而是让它们对世界状态有一致理解。

**问题表现：**
- 一个 Agent 基于过期信息行动，其他 Agent 在此基础上继续，错误会叠加放大
- 不同 Agent 使用不同工具或版本，导致输出冲突

**解决要素：**
- 共享状态
- 明确任务边界
- 仲裁机制——谁有权改什么、什么是事实 vs 推断 vs 未验证

**警告：**
没有这些约束，多 Agent 只是放大单 Agent 的混乱。

### 8. Agent 之间怎么做协调，你怎么看 AtA 这种思路？

**核心观点：**
协调应该显式而非隐式自由对话。

**显式协调的优势：**
- 输入输出清晰
- 可通过中央调度器或共享黑板实现
- 支持超时重调度、冲突解决、回滚等机制

**AtA 的风险：**
- 没有明确的访问边界和任务接收机制
- Agent 间协作可能变成"互相甩锅"或无限递归

**适用场景：**
适合能力划分，但必须有中央约束——失败后谁接手、结果归谁负责，否则"协作"变成"无人负责"。

### 9. 你怎么理解 Skill，和 Tool 的区别是什么？

**Tool：**
- 原子能力（如查数据库、读文件、发请求、执行命令）
- 最低层操作接口

**Skill：**
- 稳定的能力模块，由多个 Tool 和工作流策略组成
- 如"审阅合同风险"、"生成测试补丁"、"整理会议纪要"
- 包含任务边界、输入输出约束、错误处理逻辑
- 复用性更高

**工程视角：**
- 只暴露 Tool：模型每次重新组合，稳定性差
- 封装成 Skill：模型在更高抽象层操作，方便版本管理和效果评估

### 10. 你怎么做 Agent 的记忆？

**三种记忆类型：**

1. **短期操作记忆**：追踪当前上下文，如"上一个工具返回什么"、"当前执行到规划的哪一步"

2. **任务记忆**：存储相似任务的历史案例，如"过去案例"、"常见失败模式"

3. **长期偏好记忆**：反映用户习惯和系统配置

**注意事项：**
- 三者更新频率和读取优先级不同，混在一起会非常乱
- 记忆最好结构化，而非一股脑做成自然语言总结
- 一旦全是文本，后面检索很容易把过期信息和有效信息混在一起

**最大风险：**
记忆系统最大的风险不是"记不住"，而是"记住了不该记的"。

### 11. 如果用户输入含糊不清，你会让 Agent 追问还是自己猜？

**决策依据：任务风险和推断成本**

**低风险场景：**
- 普通问答或低风险办公任务
- 可以基于默认假设先做一步，再把假设说清楚

**高风险场景：**
- 涉及删除、发送、审批、修改正式文档等动作
- 更倾向于先追问
- Agent 一旦自己猜错，后果可能比多问一句严重得多

**原则：**
- 缺的是"可补充背景"可以先尝试
- 决定动作正确性的关键信息必须追问

### 12. 你会怎么设计 Agent 的评测体系？

**三层评估：**

1. **任务完成层**：目标是否达成

2. **过程质量层**：
   - 工具选得对不对
   - 参数对不对
   - 有没有多余步骤
   - 失败后能否自恢复

3. **系统稳定性层**：
   - 延迟
   - 重试率
   - 人工介入率
   - 高风险错误率

**回放评测：**
- 把真实脱敏任务按原始环境重新放一遍
- 观察 Agent 每一步怎么决策
- 静态 benchmark 只能说明它在题库上会做题
- 回放评测才能看出它在动态环境里会不会犯蠢

**核心观点：**
很多 Agent 看起来能干，但过程质量差，生产环境不可用。

### 13. 如果工具返回脏数据或者部分缺失，Agent 应该怎么处理？

**最糟糕的做法：**
把脏数据直接当事实继续往后推

**稳健做法：**
1. 先对工具输出做 schema 校验和置信标记
2. 把"字段缺失"、"类型异常"、"明显冲突"等情况显式暴露给上层状态机
3. Agent 看到的不是一段原始脏文本，而是一个带质量标签的 observation
4. 后续要么触发重试，要么降级，要么追问用户
5. 不至于直接把错误扩散下去

### 14. 你怎么防止 Agent 在复杂任务里越跑越偏？

**核心策略：减少不可逆错误**

**分阶段校验点：**
- 分析完确认关键信息
- 执行前校验参数/目标
- 执行后验证结果是否符合预期
- 发现偏差时回滚到最近的校验点，而不是让模型继续瞎跑

**上下文污染处理：**
- 复杂任务中每阶段结束后做局部摘要和状态压缩
- 只保留必要的当前信息
- 累积的上下文越多，越容易让模型"发懵"

## 三、xiaohongshu

### 1. 如果知识库原始数据不是 markdown，而是 PDF、PPT、图片或者网页，你会怎么处理进 RAG 系统？

**处理流程：**

1. **PDF/PPT**：提取标题、章节、表格、页码

2. **图片**：进行 OCR 识别

3. **网页**：提取正文和元数据

4. **统一清洗**：去噪、去页眉页脚

5. **分块**：按语义切分

6. **添加元数据**：来源、时间、文档层级

7. **生成 embedding 并建索引**

**收益：**
- 召回更稳定
- 来源可追溯
- 权限可控

### 2. 在 RAG 检索过程中，常用的检索机制有哪些？

**三种常见类型：**

1. **关键词检索**：适合专业术语和精确匹配

2. **向量检索**：适合语义相似但表述不同的查询

3. **混合检索**：通常更稳定，兼顾字面和语义匹配

**线上系统实践：**
- 一般会用多路召回 + 重排序
- 不是单路召回

### 3. RAG 评价指标一般怎么设计？

**双层评估：**

**检索侧：**
- 召回是否命中相关段落
- 指标：Recall@K、MRR、NDCG

**生成侧：**
- 答案相关性
- 忠实度
- 上下文利用有效性
- 检查模型是否基于召回内容回答，而非自行发挥

**注意：**
用户感知的"听起来对"是不够的，因为模型可能生成流畅但事实错误的回答。

### 4. 为什么很多场景会优先选 RAG，而不是直接微调大模型？

**原因分析：**
- 业务知识更新快，直接微调成本高、周期长，知识变化就要重新训练
- RAG 更适合私域知识、更新快、易溯源
- 微调适合风格调整、任务习惯、领域能力边界，而非频繁事实更新

**实践做法：**
RAG 和微调常常结合使用，不是二选一。

### 5. MCP 和 Function Call 的区别是什么？

**Function Call：**
- 模型在对话中通过 schema 调用预定义工具
- 关注"调用哪个函数"

**MCP：**
- 更偏向协议层
- 不仅描述函数，还描述资源、能力和上下文
- 如"上下文暴露"、"资源列表"等

### 6. 为什么 AI 应用里经常需要做两级缓存？

**原因：**
很多 AI 链路步骤（embedding、检索、重排、生成）成本高、延迟高

**两层缓存设计：**
1. **本地缓存**：进程内热点
2. **Redis**：多实例共享

**收益：**
- 减少重复 embedding、重复调用、重复生成
- 降低成本，稳定响应时间

**注意事项：**
缓存穿透、热点 key、版本失效等问题。

### 7. 如果要做一个智能客服类 AI 应用，整体链路怎么设计？

**典型流程：**

1. **问题分类**：FAQ、知识问答、工具调用、人工接管

2. **简单固定问题**：模板回复

3. **知识类问题**：RAG

4. **订单/查询/发货/账户查询**：工具调用

5. **高风险或低信任问题**：直接转人工

**核心链路必备：**
- 会话管理
- 权限校验
- 敏感词过滤
- 日志追踪
- 效果评估

### 8. 为什么有些场景会选 PGVector，而不是直接用 ES 做向量检索？

**原因：**
- 业务数据本身就在 PostgreSQL 里
- 没必要单独搭一套向量检索集群
- PGVector 集成和维护成本低
- 结构化数据和向量数据统一管理，运维简单

**ES 的优势：**
- 全文 + 向量混合检索有优势

**PGVector 适用场景：**
- 业务数据在 PG 中
- 快速部署需求

### 9. MySQL 和 Elasticsearch 的区别是什么，AI 应用里怎么分工？

**MySQL：**
- 适合事务型、关系型、强一致的主业务数据

**Elasticsearch：**
- 适合全文检索、模糊匹配、聚合分析

**AI 应用分工：**
- MySQL：存用户、配置、权限、任务记录
- ES：做文本搜索、内容检索、日志分析

**关系：**
两者不是替代关系，更多是主存储和搜索层分工。

### 10. 为什么 Elasticsearch 要使用倒排索引，而不是顺序扫描？

**原因：**
- 全文检索的数据量通常很大
- 每次查询顺序扫所有文档代价太高

**倒排索引原理：**
- 先建立"词 → 文档"的映射
- 查询时直接定位命中文档，再做过滤和打分
- 无论是关键词查询、短语匹配还是多条件组合搜索，效率都比顺序扫描高很多

## 四、alibaba

### 1. Agent 的架构设计？

**完整 Agent 的分层架构：**

1. **任务入口层**：负责接收用户问题和上下文

2. **决策层**：负责意图识别、任务拆解、规划和工具选择

3. **执行层**：真正调工具、查知识库、访问服务

4. **记忆和状态层**：管理多轮上下文、历史结果、中间变量

5. **验证层（可选）**：加执行前后校验（参数有效性、结构完整性、结果一致性）

**核心挑战：**
不是"能不能思考"，而是"能不能稳定执行"。

### 2. RAG 的检索如何实现？

**离线部分：**
- 文档清洗
- 分块
- 去重
- embedding 计算
- 向量数据库建索引

**在线部分：**
- 用户查询 → 编码成向量 → 向量库检索
- BM25/关键词混合召回
- rerank 模型重排
- 组装 top-k 相关证据给 LLM

**分块策略：**
- 分块大小、滑动窗口重叠
- 每块附带元数据（标题、章节路径、来源）

**最终效果：**
不只取决于向量，还取决于分块设计、召回策略、重排质量。

### 3. 预训练数据清洗方法？

**三类问题数据：**

1. **脏数据**：乱码、HTML 残留、脚本片段、异常符号、编码错误

2. **重复数据**：精确重复、近似重复

3. **低质量数据**：广告、无意义填充文本、模板、机翻片段、损坏文本、拼接错乱文本

**清洗步骤：**

1. **规则清洗**：去标签、过滤控制字符、长度约束、语言检测

2. **质量过滤**：困惑度过滤、分类器识别垃圾文本、关键词规则

3. **去重**：MinHash、SimHash、LSH 近似去重

**核心观点：**
真正影响模型上限的，往往不是架构，而是预训练数据质量。

### 4. 给定一个时间序列，如何通过机器学习的方法建模筛选出重要特征？

**特征转换：**

1. **统计特征**：均值、方差、最大最小、移动平均等

2. **趋势特征**：斜率、导数等

3. **波动特征**：振幅、频率等

4. **周期特征**：傅里叶周期特征、滞后值、滚动窗口特征等

**建模方法：**
使用 XGBoost、LightGBM、Random Forest 等模型训练并输出预测。

### 5. Agent 工具调用你知道怎么训练吗，训练集应该包含哪些，怎么得到训练的数据集？

**训练样本类型：**

1. **应该调用的案例**

2. **不该调用的案例**

3. **需要多工具顺序调用的案例**

**数据来源：**

1. **人工标注轨迹**

2. **线上高质量人/系统交互日志**

3. **规则合成数据**

4. **强模型自生成轨迹 + 人工校验**

**关键挑战：**
缺参数、工具空返回、多工具场景的处理优先级很重要，否则模型容易过拟合或上线后乱调。

### 6. 构造数据集遇到过什么难点，怎么解决？

**难点一：合成/真实数据不匹配**
- 人工数据太标准化，用户表述过于完整，参数过于齐全
- 上线后差距大，用户没说全意图 → 模型就崩了

**解决方案：**
- 导入真实线上 query
- 按意图、复杂度、缺参数、歧义性分类
- 针对性补齐

**难点二：标注不一致**
- 同一个问题，不同人标成不同工具/标准

**解决方案：**
先统一 schema 和决策标准。

**难点三：短尾样本少**
- 模板改写
- 对抗生成
- 人工标注边界案例
- 优先高风险失败点

### 7. LoRA 和全参数微调的区别？

**LoRA：**
- 在原权重层附近插入低秩矩阵
- 只训练少量新参数
- 冻结原模型权重

**全参数微调：**
- 直接更新所有模型参数

**LoRA 优势：**
- 内存占用低
- 训练快
- 多任务切换容易
- 适合领域适配、指令跟随、工具使用等格式对齐场景

**权衡：**
全参数微调更灵活，但计算/内存成本更高。

### 8. DPO、PPO、GRPO？

**PPO：**
- 策略梯度方法，需要奖励模型给输出打分，再用奖励更新策略
- 灵活但训练路径长，超参敏感，稳定性中等

**DPO：**
- 直接把偏好学习写成监督目标
- 用 chosen 和 rejected pair 直接优化策略
- 无需采样和轨迹长度，工程简单

**GRPO：**
- PPO 的扩展，从 pairwise 泛化到 groupwise 比较
- 利用多个候选间的相对偏好
- 天然处理排序、多样本比较，奖励结构更丰富

**核心区别：**
三者最大区别不只是公式，而是它们依赖的数据形态、训练稳定性和适用任务范围不一样。

### 9. kernel 级别的优化？

**目标：**
- 减少显存访问开销
- 减少 kernel 启动次数
- 提高并行利用率

**原因：**
很多模型跑得慢不是算力不够，而是 memory bound。

**技术手段：**
- CUTE DSL（张量操作领域特定语言）
- 手写 CUDA kernel
- 算子融合

## 五、Vibe Coding 面试题目

### 如何应对面试中的 Vibe Coding 题目？

**核心观点：**
面试官想看到的不是 AI 有多聪明，而是你作为"架构师"和"技术主导者"的思维过程。AI 只是你的高级键盘，技术选型、边界划分和兜底策略必须由你来掌控。

**对学生而言：**
不要求有真实落地结果，但一定会要求有把项目做深、做闭环的能力。

### Step 1：定义边界与骨架——展现全局观

**你的 Prompt：**
"我现在需要写一个高并发下的***，功能如下：***"

**目的：**
先把数据结构定下来，这叫"谋定而后动"。

### Step 2：选择并指挥核心模型——展现技术掌控力

**你的 Prompt：**
"为了防止内存溢出（OOM），请使用 ArrayBlockingQueue 作为底层的缓冲队列。"

**目的：**
展现你对并发工具的掌控力。AI只是你的键盘，技术选型必须是你来做。

### Step 3：攻克难点——展现深度思考

**目的：**
每个面试题都有一个"坑"或难点，通过单独抽出来让 AI 解决，能展现你对复杂场景的预判能力。

### Step 4：兜底与测试——展现工业级标准

**你的 Prompt：**
"最后，请帮我加上***，确保程序崩溃时，***。然后生成一个并发测试用例。"

**目的：**
证明这是一个达到工业级可用标准（能上线）的代码，而不是学生做的玩具。

**面试建议：**
面试中，最好将自己的 Prompt 解释出来，因为面试官看不到你脑子里的想法。

## 示例：高并发事件总线实现

### Step 1（搭架子）

**你的想法：**
定义系统职责和扩展边界，防止 AI 生成代码结构失控。

**你的 Prompt：**
"我现在需要写一个高并发场景下的本地事件总线。核心模块包括：Publisher、Subscriber 和 Topic。请先不要写任何业务逻辑，只帮我定义这三个核心类的 Interface 接口和基础数据结构骨架。"

### Step 2（定核心）

**你的想法：**
高并发场景下要考虑慢消费者问题，如果用无界队列如 LinkedBlockingQueue 可能导致消息积压和 OOM。

**你的 Prompt：**
"为了防止 OOM，请在 Topic 内部严格使用 ArrayBlockingQueue 作为底层的缓冲队列，并强制要求在初始化时传入 Capacity（容量大小）。现在请基于这个约束，实现具体的 publish 和 consume 逻辑。"

### Step 3（攻克难点）

**你的想法：**
正常流程走通只是及格，需要考虑两个极端情况：
1. 队列满时如何避免死锁或丢数据
2. 消费者抛异常时如何避免雪崩

**你的 Prompt：**
"现在处理极端场景：
1. 当 ArrayBlockingQueue 满时，请为 publish 方法实现带有超时机制的阻塞重试逻辑。
2. 如果 Subscriber 消费时抛出异常，请实现一个死信机制（DLQ）或异常隔离记录，确保消费线程不崩溃。"

### Step 4（兜底）

**你的想法：**
线上服务重启或发布时，强杀进程会导致消息丢失，必须有优雅停机机制。同时，并发代码必须通过多线程压测自证清白。

**你的 Prompt：**
"最后，为了达到工业级标准：
1. 请加上 Graceful Shutdown（优雅停机）逻辑，通过 JVM Hook 确保停止接收新消息并消费完存量消息。
2. 生成一个多线程并发压测用例（100 个线程 publish，50 个线程 consume），验证线程安全。"
